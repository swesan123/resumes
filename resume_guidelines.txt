Software Engineering Resume Bullet Point Guidelines
===================================================

High-level principles
---------------------
- Focus every bullet on **impact**, not duties. Answer: What did you do, how did you do it, and why did it matter?
- Use a consistent formula: Action Verb + What you built/changed + Technologies used + Measurable result.
- Prioritize clarity and scannability over buzzwords or dense jargon.
- Make each line independently compelling; recruiters often skim only the first few words and any bolded keywords.
- Assume a **non-engineer recruiter** is reading first; prefer simple, concrete wording over domain jargon.

Bullet point structure
----------------------
- Start with a strong action verb:
  - Examples: Architected, Designed, Implemented, Engineered, Optimized, Refactored, Automated, Deployed, Led, Spearheaded.
  - Avoid weak openings: "Responsible for", "Worked on", "Helped with", "Tasked with".
- Clearly state what you built or changed:
  - Name the system, feature, or component (API, service, pipeline, model, dashboard, CLI, etc.).
  - Briefly describe the problem or goal if it is not obvious.
- Include the key technologies and skills you want to be searched for:
  - Mention languages, frameworks, databases, cloud providers, tools, paradigms (e.g., Python, React, PostgreSQL, AWS, Docker, Kubernetes, REST, gRPC, RAG, LLMs).
- End with a concrete outcome:
  - Use metrics where possible: performance, reliability, revenue, cost, engagement, throughput, latency, accuracy, error rates.
  - If you cannot share exact numbers, use relative metrics (e.g., "reduced build time by ~40%").

Impact and metrics
------------------
- Always try to quantify impact:
  - Performance: latency (ms), throughput (req/s), CPU/memory savings, build time.
  - Reliability: error-rate reduction, incident frequency, uptime/SLA improvements.
  - Scale: number of users, requests per day, size of datasets, number of services.
  - Business: revenue, cost savings, churn reduction, conversion or activation rate.
- Example transformations:
  - Weak: "Improved API performance."
  - Strong: "Optimized Node.js REST APIs and PostgreSQL queries, cutting P95 latency from 900ms to 320ms (64% faster) for 120k+ daily requests."
  - Weak: "Worked on CI/CD."
  - Strong: "Implemented GitHub Actions–based CI/CD pipelines with Dockerized builds and automated tests, reducing deployment failures by 30% and cutting release time from days to hours."

Keyword and skills visibility (bolding strategy)
-----------------------------------------------
- Recruiters and automated screeners skim for specific skills and technologies. Make them visually obvious.
- In LaTeX, bold important keywords using \\textbf{...} so they stand out on a quick scan:
  - Languages: \\textbf{Python}, \\textbf{TypeScript}, \\textbf{Java}, \\textbf{C++}.
  - Frameworks and libraries: \\textbf{React}, \\textbf{Node.js}, \\textbf{Django}, \\textbf{TensorFlow}, \\textbf{PyTorch}, \\textbf{LangChain}.
  - Cloud and infrastructure: \\textbf{AWS}, \\textbf{GCP}, \\textbf{Azure}, \\textbf{Docker}, \\textbf{Kubernetes}.
  - Domains: \\textbf{LLM Agents}, \\textbf{RAG Systems}, \\textbf{NLP Pipelines}, \\textbf{Distributed Systems}.
- Recommended pattern inside bullets (adapt for your macros):
  - "Built a \\textbf{Python} and \\textbf{LangChain}-based RAG assistant deployed on \\textbf{AWS ECS} and \\textbf{Lambda}, cutting support response time by 45\%."
  - "Implemented a \\textbf{React} and \\textbf{TypeScript} front-end integrated with \\textbf{Node.js} and \\textbf{PostgreSQL} APIs, improving checkout conversion by 8\%."
- Be intentional: bold only the most important skills and technologies per line (usually 2–4 terms) so the page does not become visually noisy.
 - When tailoring to a specific job, **only bold skills that actually appear in the job description** or very close synonyms.

Style and phrasing guidelines
-----------------------------
- Use present tense for ongoing roles and past tense for completed roles and projects.
- Keep bullets concise: typically 1–2 lines each; avoid wrapping to a third line when possible.
- Avoid filler words and vague phrases:
  - Replace "various", "multiple", "several", "a lot of" with concrete nouns and counts.
  - Replace "helped", "assisted", "involved in" with direct ownership verbs.
- Avoid internal or confidential code names unless you briefly explain them.
- Prefer specific nouns over abstractions:
  - "payment retry service" instead of "critical microservice".
  - "log aggregation pipeline" instead of "data system".
 - **Avoid brackets** like `()`, `[]`, `{}` in visible bullets and skills; prefer simple comma-separated lists instead (e.g., `Linux, CentOS, Ubuntu` instead of `Linux (CentOS, Ubuntu)`).

Project and experience bullets (engineering-specific tips)
---------------------------------------------------------
- For backend / infrastructure work:
  - Emphasize APIs, data models, reliability, scaling, observability, and security.
  - Call out patterns and tools: \\textbf{REST}, \\textbf{gRPC}, \\textbf{tRPC}, \\textbf{Kafka}, \\textbf{Redis}, \\textbf{Prometheus}, \\textbf{OpenTelemetry}.
- For frontend work:
  - Emphasize performance, UX, accessibility, and design systems.
  - Mention frameworks and patterns: \\textbf{React}, \\textbf{Next.js}, \\textbf{Redux}, \\textbf{TypeScript}, \\textbf{Tailwind CSS}.
- For ML / data work:
  - Emphasize data pipelines, model training/evaluation, and productionization.
  - Name key tools: \\textbf{TensorFlow}, \\textbf{PyTorch}, \\textbf{scikit-learn}, \\textbf{pandas}, \\textbf{NumPy}, \\textbf{Spark}, \\textbf{LangChain}, \\textbf{RAG}.
- For DevOps / platform work:
  - Emphasize CI/CD, infrastructure as code, monitoring, and reliability.
  - Call out: \\textbf{Docker}, \\textbf{Kubernetes}, \\textbf{Terraform}, \\textbf{GitHub Actions}, \\textbf{CircleCI}, \\textbf{Jenkins}.

Common bullet templates (adaptable)
-----------------------------------
- "Implemented [system/feature] using \\textbf{[tech1]}, \\textbf{[tech2]}, and \\textbf{[tech3]}, resulting in [metric] improvement in [latency/throughput/reliability/etc.]."
- "Designed and deployed [pipeline/service] on \\textbf{[cloud]} with \\textbf{[infra tools]}, handling [scale] and reducing [cost/latency/errors] by [X\%]."
- "Built [UI/feature] in \\textbf{[frontend techs]} to enable [user capability], increasing [engagement/conversion/retention] by [X\%]."
- "Developed and productionized \\textbf{[ML/LLM]} models using \\textbf{[libraries]} and \\textbf{[cloud services]}, improving [accuracy/recall/etc.] from [A] to [B]."
- "Automated [manual process] with \\textbf{[language/tool]}, saving [N] engineer-hours per [week/month] and reducing errors by [X\%]."

How to use these guidelines with your LaTeX resume
--------------------------------------------------
- When writing or revising bullets, run through this quick checklist:
  1) Does the bullet start with a strong action verb?
  2) Does it clearly state what you built or changed?
  3) Does it include the key technologies or skills you want to highlight (with \\textbf{...})?
  4) Does it include at least one concrete metric or meaningful outcome?
  5) Is it concise (ideally 1–2 lines) and free of vague language?
- Apply consistent bolding across the resume so recruiters can quickly scan for:
  - Languages and frameworks.
  - Cloud / infra tools.
  - Specialized domains (ML, LLMs, RAG, security, distributed systems).

LaTeX editing policy (no deletions)
-----------------------------------
- When tailoring, **do not delete** lines from `.tex` files to hide content.
- Prefer **`%` comments** for skills lines, `\input{...}` project lines, sections, or old bullets so nothing is lost and variants can be restored quickly.

Single-page layout, sections, and skills alignment
--------------------------------------------------
- Default to a **single-page resume** for internship, new grad, and early-career roles:
  - Sections, in order: Heading, Education, Technical Skills, Work Experience, Projects, (Relevant Courses if space permits).
  - Omit or comment out lower-priority sections (e.g., Hobbies, extra courses) when they cause overflow.
- Treat **Technical Skills** as a promise that must be **proven in bullets**:
  - Only list skills that appear in at least one Experience, Project, or Courses bullet **in the current one-page resume**.
  - Derive the skills list **from the active sections** (current projects/experience shown), not from your entire background.
  - When tailoring, prefer **commenting out** unused skills over adding new bullets just to justify them.
  - **Formatting**: Bold **only** the category labels in each line (e.g. `\textbf{Languages}{: Python, C}`). Do **not** bold individual tools or languages after the colon—keep post-colon lists unbolded for a clean scan pattern.
- For company or project headings that include a skill list (e.g., `Company | \\textbf{Python}, \\textbf{Linux}`):
  - Ensure each bolded technology also appears in the bullets for that entry.
  - Do **not** bold tools that are not clearly supported by the underlying work.

Projects and courses
--------------------
- Projects:
  - For a given job, show **exactly 3 projects** (minimum 3, maximum 3), chosen to best match the posting’s core skills and domains.
  - Keep each project to **2 bullets** focused on what you built, how you tested it, and any scale/metrics.
  - Prefer projects that provide **“proof”** for the most important skills in the job description.
  - When a project heading includes a tech stack (e.g., `Project | Python, Docker, PostgreSQL`):
    - **Every listed technology must have at least one supporting bullet** that mentions it.
    - If a technology is not supported by a bullet, **remove it from the heading** instead of adding a weak bullet.
    - For technologies that *are* supported and important, **bold them both in the project heading and in the supporting bullet** (e.g., `\textbf{Python}` in both places).
  - Ensure every project has a **clear date range** on the heading that reflects reality.
    - For your current degree timeline, keep project dates between **September 2025 and March 2026** unless there is a good reason to do otherwise.
- Courses:
  - Prefer a **single-line “Relevant Courses”** entry of 3–6 short course names when space is tight.
  - Choose courses that directly support the job’s requirements (e.g., real-time, systems, hardware, security).
  - Keep the full course list in comments in `courses.tex` so targeted subsets can be enabled per job without retyping.

Practical one-page heuristics for this LaTeX template
-----------------------------------------------------
- For this `main.tex` layout (`11pt`, letter, current `config/formatting`):
  - A single page typically fits:
    - Heading, Education, Technical Skills, **1 work experience entry with up to 5 bullets**, 3 projects × 2 bullets, and a **one-line** Relevant Courses row.
  - If the compiled PDF overflows to a second page:
    - First, remove or merge lower-impact bullets from Projects or Experience until each entry has at most 2–3 bullets.
    - Next, comment out optional sections (Hobbies, extra courses) or long skill lists.
    - Recompile after each trimming pass rather than making many speculative edits.

Saving postings and compiling role-specific resumes
----------------------------------------------------
- When tailoring against a job description (paste or fetched URL):
  - Persist the **full posting text** under `postings/` as a **new** dated `.txt` file (see `.cursor/skills/latex-resume-tailoring/SKILL.md` for `<role>_<company>-YYYY-MM-DD.txt` slugs and optional `#` metadata headers).
  - Do **not** overwrite an existing `postings/*.txt` unless the user explicitly asks.
- After tailoring (and optional verification) for a specific company/role:
  - Compile `main.tex` to produce an updated `main.pdf`.
  - Export the PDF using a **role-and-date-based basename**:
    - Pattern basename: `<Role-With-Dashes>-YYYY-MM-DD.pdf`
    - Example basename: `Validation-Engineer-1-2026-05-01.pdf`
  - **Destination directory**:
    - When the employer is known, resolve `<CompanyDir>` (e.g. `AMD`, `Wealthsimple`, `Capital One`, `Canonical`)—reuse an existing folder name when it already matches on disk.
    - Create the folder if needed: **`mkdir -p resumes/<CompanyDir>`**, then save **`resumes/<CompanyDir>/<Role-With-Dashes>-YYYY-MM-DD.pdf`**.
    - When the employer is **unknown**, save **`resumes/<Role-With-Dashes>-YYYY-MM-DD.pdf`** at the **repo root** `resumes/` level only (do not invent a company folder name).
  - Convert spaces in the role portion to dashes (`-`) so filenames are shell- and URL-friendly.

Verification pass and match scoring
-----------------------------------
- After tailoring the resume to a specific job posting, run a **verification pass** before considering it final.
- Use up to **3 lightweight passes** over the job description and resume:
  1) **Job extraction pass**
     - Re-extract:
       - Role/title and level.
       - 8–12 core skills/keywords from the posting (languages, tools, domains, OS, etc.).
       - 3–5 core responsibilities (what this role actually does day to day).
  2) **Section-by-section alignment pass**
     - For each section (Technical Skills, Work Experience, Projects, Relevant Courses):
       - Confirm that the **most important JD skills** appear somewhere on the page, ideally in bullets.
       - Check that any **bolded skills** actually appear in the job description or are very close synonyms.
       - Flag any bolded technologies that are not clearly supported by the job description or experience.
       - Ensure there are **no brackets** in visible text and that bullets follow the style rules above.
  3) **Style and layout pass**
     - Confirm:
       - One-page output for this template.
       - Exactly 3 projects and at most 2 bullets per project.
       - Relevant Courses is a single line with 3–6 short course names.
       - Each bullet is 1–2 lines with at least one concrete outcome or scale detail where reasonable.
- Compute a simple **match score (0–100)**:
  - Example breakdown:
    - 40 points: coverage of top JD skills/keywords.
    - 30 points: coverage of core responsibilities (what the role actually does).
    - 30 points: adherence to these guidelines (style, bolding, layout, no brackets).
- If the score is **below ~85**, attempt targeted improvements:
  - Tighten or swap bullets so they speak more directly to the JD.
  - Swap in better-aligned projects or courses if available.
  - Fix bolding (remove noisy bold, add missing bold for key JD skills where supported by experience).

Missing skills tracking
-----------------------
- To decide what to learn next, maintain a simple skill counter in `missing_skills.txt`:
  - When verifying against a job posting, build the set of **important JD skills/keywords**.
  - For each such skill that does **not** appear in your current one-page resume (Experience, Projects, Courses, Technical Skills):
    - Add it to `missing_skills.txt` or increment its count if it already exists.
  - Do **not** add skills that you already know and simply chose not to highlight for this job.
- Over time, use the highest-count entries in `missing_skills.txt` as a priority list for:
  - What technologies or domains to study.
  - Which kinds of projects or coursework to seek out to close the gaps.
